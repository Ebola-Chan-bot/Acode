## 范围

本文只统计从基准提交到当前工作树 **最终保留下来的净变更**。中途出现但已经被回退的纯诊断日志、临时探针、局域网调试辅助修改，不计入“最终净变更”，只在“调试性变更检查”中说明。

## 总结结论

这段时间的净变更主体是一次较大的终端子系统修复与增强，核心目标有四个：

1. 让 Alpine/AXS 终端环境能够自动下载、安装、启动、修复。
2. 让终端会话和 UI 设置更稳定，包括会话恢复、字体/字距/图像/连字等配置。
3. 解决 proot/AXS 相关的已知高频故障，包括 HTTP 探活不准、PTY 打不开、seccomp 导致崩溃、shebang 在 proot 下异常等。
4. 给 Android 侧执行器补齐后台下载、前台保活和更可靠的服务绑定能力。

从净结果看，**没有保留明显的纯日志诊断性改动**，此前为了定位问题而加过的 AXS/PT Y/proot 调试日志已经不在当前净差异中。当前仅剩少量正常错误处理日志和一处命令缺失绕过逻辑，这些都是功能性或故障处理的一部分，不属于“纯调试残留”。

需要单独关注的非核心净变更有两个：

1. [package-lock.json](package-lock.json) 相对基准提交存在少量锁文件元数据差异，主要体现为若干包的 `peer` 标记变化，信号较低，需要和真正的功能变更分开看待。
2. [src/plugins/system/www/plugin.js](src/plugins/system/www/plugin.js) 的 `in-app-browser` 回调区块在净差异里主要是格式波动，真正功能性新增只有 `copyAsset` 配套接口。

## 一、终端安装与启动链路

### 相关文件

- [src/plugins/terminal/www/Terminal.js](src/plugins/terminal/www/Terminal.js)
- [src/components/terminal/terminal.js](src/components/terminal/terminal.js)
- [src/plugins/terminal/src/android/Executor.java](src/plugins/terminal/src/android/Executor.java)
- [src/plugins/terminal/src/android/BackgroundExecutor.java](src/plugins/terminal/src/android/BackgroundExecutor.java)
- [src/plugins/terminal/src/android/DownloadHelper.java](src/plugins/terminal/src/android/DownloadHelper.java)
- [src/plugins/terminal/www/Executor.js](src/plugins/terminal/www/Executor.js)
- [src/plugins/system/android/com/foxdebug/system/System.java](src/plugins/system/android/com/foxdebug/system/System.java)
- [src/plugins/system/www/plugin.js](src/plugins/system/www/plugin.js)

### 净功能变更

1. 终端安装从“每次粗暴重装”变成了“分阶段缓存安装”。
   - [src/plugins/terminal/www/Terminal.js](src/plugins/terminal/www/Terminal.js) 现在将安装拆成下载、解压、配置三个阶段，并使用 `.downloaded`、`.extracted`、`.configured`、`.download-manifest` 等标记做增量复用。
   - 行为影响：重复安装或重开终端时，不再总是重新下载 Alpine 和 AXS，网络和等待时间明显下降。

2. Android 执行器新增原生下载能力，替代旧的前端 HTTP 下载路径。
   - 新增 [src/plugins/terminal/src/android/DownloadHelper.java](src/plugins/terminal/src/android/DownloadHelper.java)，并在 [src/plugins/terminal/src/android/Executor.java](src/plugins/terminal/src/android/Executor.java) 与 [src/plugins/terminal/src/android/BackgroundExecutor.java](src/plugins/terminal/src/android/BackgroundExecutor.java) 暴露 `download` action。
   - [src/plugins/terminal/www/Executor.js](src/plugins/terminal/www/Executor.js) 对应新增 `download(url, dst, onProgress)`。
   - 行为影响：下载支持原生重定向跟随、超时控制和进度回调，更适合大文件获取。

3. 调试构建下支持用内置资产覆盖 `axs`。
   - [src/plugins/terminal/www/Terminal.js](src/plugins/terminal/www/Terminal.js) 新增 `refreshAxsBinary()`，依赖 [src/plugins/system/android/com/foxdebug/system/System.java](src/plugins/system/android/com/foxdebug/system/System.java) / [src/plugins/system/www/plugin.js](src/plugins/system/www/plugin.js) 新增的 `copyAsset` 接口。
   - 行为影响：debug 构建可以确保手机里实际运行的 `axs` 与当前打包产物一致，避免“脚本/前端更新了但旧二进制还在跑”的错配。

4. AXS 启动完成检测从“只看 PID 活着”改成“探测 HTTP 端点真的可用”。
   - [src/components/terminal/terminal.js](src/components/terminal/terminal.js) 在创建终端时加入 `pollAxs()`，通过 `http://localhost:8767/` 探活，而不是单纯 `kill -0`。
   - 解决问题：进程存在但 HTTP 服务尚未就绪时，旧逻辑会误判成功，随后马上在创建终端或 WebSocket 建连阶段失败。

5. 启动失败后支持自动修复，而不是只弹失败提示。
   - [src/components/terminal/terminal.js](src/components/terminal/terminal.js) 中，当 AXS 启动后 HTTP 仍不可达，会触发一次 repair 流程：停止旧进程，重新执行 `startAxs(true)` 进行配置修复，再重启。
   - 修复后若仍失败，还会调用 `resetConfigured()` 清掉 `.configured`，让下次重新走完整安装。
   - 解决问题：终端环境部分损坏时，不必手动清目录后重装。

6. PTY 打开失败时支持自动刷新二进制并重试一次。
   - [src/components/terminal/terminal.js](src/components/terminal/terminal.js) 在 `/terminals` 返回 `Failed to open PTY` 时，会刷新 `axs` 资产、重启服务并重试创建终端。
   - 解决问题：二进制不兼容或旧版 `axs` 残留导致的 PTY 打不开问题。

7. 卸载被拆分为“保留下载缓存的卸载”和“完整卸载”。
   - [src/plugins/terminal/www/Terminal.js](src/plugins/terminal/www/Terminal.js) 中 `uninstall()` 现在仅删除 rootfs 和配置标记，保留下载缓存；新增 `uninstallFull()` 做全清。
   - [src/settings/terminalSettings.js](src/settings/terminalSettings.js) 相应加入“是否同时删除下载缓存”的确认逻辑。
   - 行为影响：普通重装更快，用户可以选择保留或清空缓存。

### 解决的问题

- AXS 已启动但 HTTP 端口未就绪，终端误判可用。
- rootfs 或配置局部损坏后只能手动删除目录重装。
- 调试构建经常跑到旧版 `axs`。
- PTY 初始化失败后没有自动恢复路径。
- 重装总是重复下载，成本过高。

### 潜在风险
- 对比对象：当前状态

## 二、proot 兼容性与 shell 侧绕过

### 相关文件

- [src/plugins/terminal/scripts/init-alpine.sh](src/plugins/terminal/scripts/init-alpine.sh)
- [src/plugins/terminal/scripts/init-sandbox.sh](src/plugins/terminal/scripts/init-sandbox.sh)

### 净功能变更

1. 安装阶段对 bash / wget / tzdata / command-not-found 做更稳妥的存在性检查和补装。
   - [src/plugins/terminal/scripts/init-alpine.sh](src/plugins/terminal/scripts/init-alpine.sh) 中改为通过文件和 APK 数据库检查，失败后带镜像重试，并补做 `/bin/bash` 软链和 `/etc/shells` 修正。

2. `.configured` 标记改由 JS 层创建，不再在 shell 脚本里 `mkdir -p`。
   - 这是为了解决 bind-mount/Java `mkdirs()` 观察到“目录已存在”而误判失败的问题。
3. `acode` CLI 从独立 shebang 脚本改为 bash function。
   - [src/plugins/terminal/scripts/init-alpine.sh](src/plugins/terminal/scripts/init-alpine.sh) 中删除了原来的 `#!/bin/bash` 脚本生成逻辑，改为在 `initrc` 内定义 `_acode_open()` / `acode()`。
   - 解决问题：在 proot 下直接执行 shebang 脚本会触发解释器链异常，函数形式可以完全避开这条路径。

4. 新增 `command_not_found_handle()`，显式用 `/bin/sh` 执行 `/usr/libexec/command-not-found`。
   - 这是当前状态中的功能性修复。
   - 目的已在 [src/plugins/terminal/scripts/init-alpine.sh](src/plugins/terminal/scripts/init-alpine.sh) 注释中写明：绕过 proot 下 shebang 直接执行导致的 `bad interpreter: Bad address`。
   - 行为影响：输入不存在命令时，Bash 仍能调用原脚本，但不再走坏掉的 shebang 自动解释器路径。
5. 关闭 proot 的 seccomp 过滤。
   - [src/plugins/terminal/scripts/init-sandbox.sh](src/plugins/terminal/scripts/init-sandbox.sh) 新增 `PROOT_NO_SECCOMP=1`，并写明这是为了避免部分 Android 内核下的 SIGSEGV / SIGBUS。

6. 去掉 `--sysvipc`。
   - [src/plugins/terminal/scripts/init-sandbox.sh](src/plugins/terminal/scripts/init-sandbox.sh) 中明确删除该参数，并备注原因是某些内核上会触发 Bus Error。

7. AXS 启动命令增加 `--allow-any-origin`。
   - [src/plugins/terminal/scripts/init-alpine.sh](src/plugins/terminal/scripts/init-alpine.sh) 中最终执行改为 `"$PREFIX/axs" --allow-any-origin -c "bash --rcfile /initrc -i"`。
   - 解决问题：WebView 发往 localhost 的 HTTP 探活/终端请求被同源策略误伤，出现 `TypeError: Failed to fetch`，进而引发错误的 repair/reinstall 循环。

### 解决的问题

- proot 下 shebang 直接执行不稳定，影响 `acode` CLI 和 `command-not-found`。
- 某些 Android 内核上 seccomp / sysvipc 导致 proot 直接崩溃。
- WebView 到 localhost 的请求被 CORS 拦截，表现为 AXS 明明在监听但前端仍判定失败。

### 潜在风险
## 七、最终结论

1. `command_not_found_handle()` 只修复“命令不存在时的辅助脚本”这一条路径，并不等于全局修复所有 shebang 脚本。
2. 关闭 seccomp 和移除 sysvipc 是有意识的兼容性取舍，会牺牲少量能力，但从当前场景看是合理的性能/稳定性优先选择。

   - [src/components/terminal/terminal.js](src/components/terminal/terminal.js) 能在 WebSocket 输出中识别 `Error relocating` / `symbol not found`，通过 `onCrashData("relocation_error")` 上报。
   - [src/components/terminal/terminalManager.js](src/components/terminal/terminalManager.js) 对该 crash reason 做了自动卸载 rootfs、重新安装并重连的原地修复。
   - 解决问题：Alpine libc/readline 级损坏时，不再只能手动重置终端环境。

6. 终端 resize 和挂载细节增强。
   - [src/components/terminal/terminal.js](src/components/terminal/terminal.js) 与 [src/components/terminal/terminalManager.js](src/components/terminal/terminalManager.js) 里都有更稳妥的 fit / ResizeObserver / 移动端触摸选择配合处理。

7. 终端内 OSC 7777 打开文件/目录能力继续保留并配合 `acode()` 使用。
   - [src/components/terminal/terminal.js](src/components/terminal/terminal.js) 处理 `OSC 7777`。
   - [src/components/terminal/terminalManager.js](src/components/terminal/terminalManager.js) 将其接到 `openFile` / `openFolder`。

### 解决的问题

- 终端设置默认值散落，用户输入非法值时容易污染运行时。
- 应用重启后终端会话全部丢失。
- 终端初始化失败后容易留下无法交互的坏 tab。
- rootfs 发生 relocation 错误后缺少自动恢复机制。

### 潜在风险

1. 会话恢复建立在 PID 和 server 状态可信的前提上，如果外部状态漂移，恢复仍可能失败。
2. 自动 crash 修复会清空当前 rootfs，虽然这是预期行为，但一旦误判，代价较高。

## 七、最终结论

从基准提交到当前工作树，绝大多数净变更都有明确的功能目标，且核心收益集中在终端可用性与自动修复能力上。保留下来的主线价值是：

1. 终端安装、启动、检测、修复更自动化，减少手工介入。
2. 终端在 Android/proot 场景下的稳定性显著增强，尤其是 HTTP 探活、PTY 失败恢复、seccomp/sysvipc 兼容处理。
3. 会话恢复、设置项、字体和显示行为更完整。
4. 插件重载与认证日志行为更合理。

当前没有发现应当立即清理的纯调试残留。真正需要额外人工判断的只有两项低信号内容：

1. [package-lock.json](package-lock.json) 中存在低信号锁文件元数据差异，但不应直接解读成明确的功能性依赖调整。
2. [src/plugins/system/www/plugin.js](src/plugins/system/www/plugin.js) 中不伴随明确行为变化的格式波动。

除此之外，其余净变更都能对应到明确的问题修复或功能增强。
